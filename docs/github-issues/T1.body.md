## Ziel

Von monolithischem **`supabase-complete.sql`** (und analog **`supabase-license-portal.sql`**) zu **Supabase CLI-Migrations** (zeitlich geordnete Dateien unter `supabase/migrations/`):

- **Eine Baseline-Migration** mit dem aktuellen Schema (einmalig groß).
- **Künftige Schemaänderungen** nur als **kleine Deltas** (Reviews, klarer Rollout).
- **Zwei Supabase-Projekte** = zwei getrennte Migrationspfade (Mandanten-DB ≠ Lizenzportal).

**Hinweis:** Bringt **keine** messbare App-Laufzeit-Verbesserung; Nutzen = Review, Historie, kontrollierte Deployments (siehe Strategie-Dokument).

## Akzeptanz (Auszug)

- [ ] Supabase CLI im Repo etabliert (`supabase init` o. Ä.), Ordner `supabase/migrations/` für die **Haupt-App/Mandanten-DB**
- [ ] Baseline-Migration erstellt; auf **bestehenden** Prod/Staging-DBs: Historie **`repair` / applied** ohne erneutes Voll-Replay des Monolithen (siehe Strategie)
- [ ] **Lizenzportal**-Projekt: eigenes `supabase/` bzw. Migrationen unter z. B. `supabase-license-portal/supabase/migrations/`
- [ ] `supabase-complete.sql` / Lizenzportal-SQL weiterhin als **Referenz** dokumentiert (oder Regel: nur noch aus Migrationen ableiten)
- [ ] Optional: CI-Schritt (Lint / `supabase db lint`) auf `migrations/`
